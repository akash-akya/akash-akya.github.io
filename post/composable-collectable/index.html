Composable Collectable

What is composable collectable?

If you are not familiar with with collectable checkout about it HexDocs.

> If the functions in Enumerable are about taking values out, then Collectable.into/1 is about collecting those values into a structure.

Another way of thinking about these is that Enumerable helps with the act of pulling values out of something. And Collectable helps with the act of pushing values into a something. In both of these case We don't really care what implementation details of that "the something". The protocols are about the act of pulling and pushing. The something might be a list, map, tcp connection or file on disk.

Why we need collectable to be composable?

For the same reason why we need `Enum` and `Stream` module to work with Enumerable. Currently there is no equivalent modules to compose Collectable values.

Let's take an example and see

You have infinite stream of logs, each line in stream can be of different level – info, warn, error. The task is that you want to push logs to somewhere. And this somewhere is unknown and should be configurable. You might want to push all logs to s3 or push to a file. or push error logs to console and skip remaining (push to void).

How would you model this keeping the flexibility in mind? Assume that flexibility part of the requirement.

Cases like these are why and hopefully at the end of the post we solve this better using composable collectable, and convince you about the usefulness of it.

Push Containers

To avoid getting lost in the details of Collection protocol. Which just a one way to represent push container. Lets build our own push contianer in a bottom-up approach.

So we want something where we can push stuff. We don't care about implementation details of something.

Lets build this using functions. So the contract is – we call `push` function with the something as first argument and the thing we want to push as second argument. So if something wants to a push container it only has to implement `push(something, anything)` function.

# string container
def push(string, value) when is_binary(string) do
  string <> value
end

# list container
def push(list, value) when is_list(list) do
  list ++ [value]
end

# void container
def push(nil, _) do
  nil
end

Lets see how this can be used

def generate_strings(container, count) do
  chars = ?a..?g

  Enum.reduce(1..count, container, fn _, container ->
    string = to_string(Enum.shuffle(chars))
    push(container, string)
  end)
end


Foo.generate_strings("", 10) |> IO.inspect(label: "string")
Foo.generate_strings([], 10) |> IO.inspect(label: "list")
Foo.generate_strings(nil, 10) |> IO.inspect(label: "nil")

Push container does not have to be a data strucutre like list, string It can be a process, socket or file. For resource like push container We also need a way to initialise this and and release it. For example for file initialise step which is opening the file and release step might be closing the file. While at it lets group all these 3 functions into a module so it is easier to build container for different type.

defmodule PushContainer do
  defstruct [:acc, :init, :func, :finish]

  def build(init, func, finish) do
    %__MODULE__{init: init func: func, finish: finish}
  end

  def init(%PushContainer{init: init} = pc) do
    %PushContainer{pc | acc: init.(), init: :done}
  end

  def push(%PushContainer{acc: acc, push: push} = pc, value) do
    %PushContainer{pc | acc: push.(acc, value)}
  end

  def push(%PushContainer{acc: acc, finish: finish} = pc) do
    finish.(acc)
  end
end

# Lets make changes to generate strings to adopt to new interface

def generate_strings(%PushContainer{} = pc, count) do
  chars = ?a..?g

  pc = PushContainer.init(pc)

  pc =
    Enum.reduce(1..count, pc, fn _, container ->
      string = to_string(Enum.shuffle(chars))
      PushContainer.push(pc, string)
    end)

  PushContainer.finish(pc)
end

Lets build push containers using new interface


string_pc =
  PushContainer.build(
    fn -> "" end,
    fn acc, val -> acc <> str end,
    fn acc -> acc end
  )

list_pc =
  PushContainer.build(
    fn -> [] end,
    fn acc, val -> acc ++ [val] end,
    fn acc -> acc end
  )

file_pc =
  PushContainer.build(
    fn -> File.open!("somefile.txt") end,
    fn file, val -> IO.binwrite!(file, val) end,
    fn file -> File.close(file) end
  )


str = generate_string(string_pc, 10)
list = generate_string(list_pc, 10)
:ok = generate_string(file_pc, 10)

Now that we have different push containers. Let's head back to composability aspect.

[C] Composing push containers

In the case above when we call generate_string(..., 1000) we are pushing string to the container. Now say we also want to upcase the string before we push. How to do this while keeping existing code intact?

Lets create a function which wraps func with the given function. It is same as Enum.map but for push-container. Basically we need to create a new push container which wraps the passed container using the wrapper function passed.

@spec map((any -> any), PushContainer.t()) :: PushContainer.t()
def map(%PushContainer{} = pc, func) do
  PushContainer.build(
    fn -> PushContainer.init(pc) end,
    fn acc, elem -> PushContainer.push(pc, func.(elem)) end,
    fn acc -> PushContainer.finish(pc) end,
  )
end

string =
  generate_strings(
    map(string_pc, &String.upcase/1),
    10
  )

# string -> upcase -> encode64 -> create-string
string =
  generate_strings(
    map(
      map(
        string_pc,
        &Base.encode64/1
      ),
      &String.upcase/1
    ),
    10
  )

This works but as you can see in the last example it not easy to read. It is even worse if we write it in single line

generate_strings(map(map(string_pc, &Base.encode64/1), &String.upcase/1), 10)

# for comparison, how something similar would look using Enum
# Enum.map(Enum.map(generate_strings(10), &String.upcase/1), &Base.encode64/1)

What is the issue here?

The issue is that we are trained to read code top-to-bottom. Because that's how most programming languages model control flow. It is hard to simulate operation execution if this is not true. This why in Elixir we have pipe operator |> which is basically a macro-magic we use to trick our brain. This makes chaining Enum more readable. This works because in case of Enum because first argument is essentially "input" in case of the enum. The input values are pulled from the first argument, that is line before pipe which makes movement of values seem natural - top-to-bottom.

Enum.map(Enum.map(generate_strings(10), &String.upcase/1), &Base.encode64/1)

# using pipe it becomes

generate_strings(10)
|> Enum.map(&String.upcase/1)
|> Enum.map(&Base.encode64/1)

It is more obvious if we visualise the way we read the code

We can't take advantage of the pipe for push-containers because a push-container is not an input, it is where where operation push the output. So using pipe for push container appear confusing

generate_strings(map(map(string_pc, &Base.encode64/1), &String.upcase/1), 10)

# using pipe becomes

string_pc
|> map(&Base.encode64/1)
|> map(&String.upcase/1)
|> generate_strings(10)

are good at reading (or parsing code) from left to right, top to bottom. Because that's how most languages are structured and we are trained.

Let's implement few more basic operations and bundle them as Coll, sort of Collectors, similar to Enum is for Enumerable.

def Coll do
  def reduce(func, %PushContainer{} = pc) do
    PushContainer.new(
      fn ->
        PushContainer.init(pc)
      end,
      fn elem, pc ->
        func.(elem, pc)
      end,
      fn pc ->
        PushContainer.finish(pc)
      end
    )
  end

  # let's simplify map using reduce
  def map(func, %PushContainer{} = pc) do
    reduce(
      fn elem, pc ->
        PushContainer.call(pc, func.(elem))
      end,
      pc
    )
  end

  def filter(func, %PushContainer{} = pc) do
    reduce(
      fn elem, pc ->
        if func.(elem) do
          PushContainer.call(pc, elem)
        else
          pc
        end
      end,
      pc
    )
  end
end

Equipped with these functions we can compose them however we want

def format_string(label, collect) do
  Coll.map(
    fn str -> "#{DateTime.utc_now()} : #{str}" end,
    collect
  )
end

list =
  generate_strings(
    format_string(list_collect())
  )

# filter strings with length > 3 and write upcase of those strings
string =
  generate_strings(
    Coll.filter(
      fn str -> String.length(str) > 3 end,
      Coll.map(
        fn str -> String.upcase(str) end,
        string_collect()
      )
    )
  )
# => "FOO BAR"

All our functiosn works with Collect type but since elixir already provides Collectable protocol and many types and different libraries already implement it. All we need now is just to bridge the Collect and Collectable protocol for interoperability.

def from_collectable(collectable) do
  Collect.new(
    fn ->
      Collectable.into(collectable)
    end,
    fn {acc, collector_func}, elem ->
      collector_func.(acc, {:cont, elem})
    end,
    fn {acc, collector_func}, status ->
      collector_func.(acc, status)
    end
  )
end

defimpl Collectable, for: Collect do
  def into(%Collect{} = collect) do
    collector_func = fn
      acc, {:cont, elem} ->
        Collect.call(acc, elem)

      acc, :halt ->
        Collect.finish(acc, :halt)

      acc, :done ->
        Collect.finish(acc, :done)
    end

    acc = Collect.init(collect)
    {acc, collector_func}
  end
end

Example

mapset =
  Enum.into(
    [1..100],
    Coll.map(
      fn num -> num * 2 end,
      Coll.from_collectable(MapSet.new())
    )
  )


# => [2, 4, 6, 8 ...]

And for the original example problem, we set out to solve. Which is how to split stream of logs

:ok =
