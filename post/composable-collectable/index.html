Composable Collectable

Enumerable and Collectable

Often we have to work with a collection of multiple values. Maybe we need multiple values as input to the logic we are going to implement and/or our logic might produce multiple values as output. The collection of multiple values can be a list of numbers, a map of key-value pairs, or a mapset of strings. Here list, map, mapset act as a container which group or hold each element together. Each element can be a number, string, or another container. Most of the time when we face such scenarios where we need to work with multiple values, the exact type of the container is less relevant. If we are going to implement a function to sum a collection of values, we would prefer if we work any container right? because whether the container is a list or set is not relevant to "adding elements" logic. We want to separate something which represents container aspect from an instance of container. Example for an instance of container is list, set, map, etc. Want to define our operations as generic operations which work with any instance of the container. Sum operation should work with a list or set.

If we think of a generic function that needs to work with containers. there are two places where we need this container glue.

  • function might need N number values as input and need a container abstraction from where it can pull values

  • function might produce N number values as output and need a container abstraction to where it can push values

For the first type of glue we have Enumerable in Elixir. If we define a function that accepts enumerable as an argument then the caller can pass list, mapset, stream or anything which implements Enumerable protocol. Our function does not care about the type of container. All we need to know is how to get elements out of the container which is what the enumerable protocol is about.

# passing List as an argument
list = Enum.filter(1..10, fn n -> Integer.is_even(n) end)

# passing MapSet as an argument
list = Enum.filter(MapSet.new([1..10]), fn n -> Integer.is_even(n) end)

Another major advantage of enumerable is Enum and Stream modules. These modules cover almost all basic operations we might need. Among other things these act as shared vocabulary between reader and writer. And If someone is starting to learn Elixir, they don't even need to know about Enumerable protocol or even about Protocol concept itself! they can just use Enum or Stream functions and assume that these modules take list, map as an argument, and does something.

If we look at the Enum and Stream functions, most of them follow a similar spec.

@spec map(Enumerable.t(), (element() -> any())) :: list()
@spec filter(Enumerable.t(), (element() -> as_boolean(term()))) :: list()
@spec reduce(Enumerable.t(), acc(), (element(), acc() -> acc())) :: acc()
@spec split_with(Enumerable.t(), (element() -> as_boolean(term()))) :: {list(), list()}

They take "anything" which implements enumerable and do certain operation and returns predefined type, which is generally a new list for Enum and opaque Stream data type for Stream functions.

What about the second type of glue? which is about collecting values in a specific container. We want to be able to define our logic which produces multiple values in a generic way such that the caller should decide the type of container they want. For example, I should be able to define a function that generates strings, and I want to return it as a list, or set, or, as a stream. The caller of the function should be able to decide the return type of the function.

We have Collectable for this

# returns a string
Enum.into([1..10], "")

# returns a mapset
Enum.into([1..10], MapSet.new())

But the catch is we don't have standard library functions to compose, build, manipulate collectables in Elixir. Similar to Enum and Stream for Enumerable.

Why do we need composability?

Let's take an example, We have a stream of logs from multiple apps and we are tasked to write these logs to a different destination depending on the scenario. We might want to just write it to console, or a file, or s3 file system. Since the destination need to be configurable we opt to use collectable protocol so that we can plug different type of destination

It can look like this

@spec write_logs(Enumerable.t(), Collectable.t()) :: :ok
def write_logs(stream, destination) do
  stream
  |> Stream.map(fn line -> DateTime.utc_now() <> " : " <> line end)
  |> Stream.into(destination)
  |> Stream.run()
end

Then we can use it like this

# to write it to console
:ok = write_logs(stream, IO.stream())

# to write it to file
:ok = write_logs(stream, File.stream!("app.log"))

# or anything which implements Collectable protocol
:ok = write_logs(stream, whatever())

As you can see caller decides where to write. As long as the destination implements collectable protocol.

This works fine. But what if we want to buffer lines before writing to file or s3 for performance reasons?

This is where we miss the lack of composability. For Enumerable we have Stream.chunk_every and we don't have functions like that or the basic building block (like reduce) for Collectable. We can get into the details of collectable protocol to achieve this, but this will be hard to work with in the long run as we will be having more and more things to compose, like say we need to filter logs of a specific app and only write those etc.

So let's just implement all basic operations like chunk_every directly using Collectable.t()?

We can do that, that's one way of doing it. But let's try to address this from the other end. Let's build a small primitive we can pass around and then come to collectable protocol. Because I think there are interesting parallels between enumerable and collectable which is more apparent if we build it from the bottom-up.

Push Container

So let's assume there is no collectable protocol and we need to build a new abstraction. What we need is a container abstraction where we can push values.

Let's start with a simpler example, we want to create a function which generates strings and we need the caller to have control over the underlaying container type.

It can look something like this

def generate_strings(container) do
  container = push(container, "foo ")
  container = push(container, "bar")

  finish(container)
end

And we want to use it like this

container = build_string_container()
string = generate_strings(container)
# "foo bar"

container = build_list_container()
list = generate_strings(container)
# ["foo ", "bar"]

To recap, generate_strings is not tied to the implementation of the container. And it only needs to know how to push elements to the container and how to get the final value when it is done.

Implementation of push and finish can be

def push({acc, finisher}, elem) do
  {acc ++ [elem], finisher}
end

def finish({acc, finisher}) do
  finisher.(acc)
end

def build(finisher) do
  {[], finisher}
end

An interesting thing to observe here is that when we build a push container we won't be knowing from where it is going to get values pushed. We can pass it around as something where someone can push values. But that decision is not made when we build a push container. This is exactly the opposite of Enumerable. With the enumerable, we don't know where values are going to get pulled out. It is something we can pass around from where someone can pull values out.

Implementation of different builders

# builder for string
def build_string_container() do
  finisher = fn acc ->
    String.join(acc)
  end

  build(finisher)
end

# builder for list
def build_list_container() do
  finisher = fn acc -> acc end

  build(finisher)
end

We are using List to accumulate values. While this works for smaller collections but if we are working with large collections this will cause memory issues similar to how Enum can cause memory issues to Enumerable. And the list is not always the correct type to collect values. For Enumerable we have Stream which solves this issue by being lazy. That is it generates values on-demand instead of producing values in advance and storing intermediate values. In this case, we can do a similar thing. No need to store the intermediate values. We can just push that responsibility to type as soon as we get a push. Which is sort of the exact opposite of Stream in a way. Stream is memory efficient by being lazy and here push-container is memory-efficient by being eager!

Let's create an interface for the eager push. While at it, let's also give it a cleaner interface by creating a module

defmodule Collect do
  defstruct [:acc, :func, :after_func]

  alias __MODULE__

  def build(acc, func, after_func) do
    %Collect{acc: acc, func: func, after_func: after_func}
  end

  def call(%Collect{acc: acc, func: func} = collect, elem) do
    acc = func.(acc, elem)
    %Collect{collect | acc: acc}
  end

  def finish(%Collect{acc: acc, after_func: after_func} = collect) do
    after_func.(acc)
  end
end

We can use our new interface like this

def generate_strings(%Collect{} = collect) do
  collect = Collect.call(collect, "foo")
  collect = Collect.call(collect, "bar")

  Collect.finish(collect)
end

string = generate_strings(string_collect())
list = generate_strings(list_collect())

We are no longer using list anywhere and we push elements to the container as they arrive

Let's update the string implement

def string_collect() do
  Collect.build(
    "",
    fn acc, string ->
      acc <> string
    end,
    fn acc ->
      acc
    end
  )
end

We can do even better by using iolist which is much more efficient

def string_collect() do
  Collect.build(
    [],
    fn acc, string ->
      [acc | string]
    end,
    fn acc ->
      IO.iodata_to_binary(acc)
    end
  )
end

In the above examples, we are using Collect to return a value. But that's not always necessary. Collect is something where we can push values. What we do with that value is left to the type. A file can be a push container! This file push container can write to a file whenever it gets a value. And we can pass it around as value to different functions as something where different functions can write.

For resource type containers like file, we also need a callback to init the resource, like opening file for file resource before we start pushing values.

Let's update our interface and build a file container

defmodule Collect do
  defstruct [:state, :acc, :init_func, :func, :finish_func]

  alias __MODULE__

  def build(init_func, call_func, finish_func) do
    %Collect{state: :init, init_func: init_func, func: call_func, finish_func: finish_func}
  end

  def init(%Collect{state: :init, init_func: init_func}) do
    init_acc = init_func.()
    %Collect{collect | state: :started, acc: init_acc}
  end

  def call(%Collect{state: :started, acc: acc, func: func} = collect, elem) do
    acc = func.(acc, elem)
    %Collect{collect | acc: acc}
  end

  def finish(%Collect{state: :started, acc: acc, finish_func: finish_func} = collect, status) do
    after_func.(acc, status)
  end
end

With this now we have callbacks to manage init, call, and finish life-cycles of resources self-contained.

Let's implement different containers with this new interface

def string_collect do
  Collect.build(
    fn -> "" end,
    fn acc, string -> [acc | string] end,
    fn acc, _status -> IO.iodata_to_binary(acc) end
  )
end

def mapset_collect do
  Collect.build(
    fn -> MapSet.new() end,
    fn set, string -> MapSet.put(set, string) end,
    fn set, _status -> set end
  )
end

def list_collect do
  Collect.build(
    fn -> [] end,
    fn list, string -> [list | string] end,
    fn list, _status -> Enum.reverse(list) end
  )
end

def file_collect(path) do
  Collect.new(
    fn -> File.open!(path, [:write, :binary]) end,
    fn elem, file ->  IO.binwrite!(file, elem) end,
    fn file, _status ->  :ok = File.close(file) end
  )
end

Now that we have different push containers. Let's head back to composability aspect.

How to compose different push containers?

Say we want to upcase all the strings before we write to a file. We don't know where or who is going to push (write) to this file since the push-container is something we can pass around different functions, but we must upcase before we write. Of course, we can build a new push container for this specific case using Collect.new but it would be brittle. We want to be able to separate this upcase logic with the specific type details. So that if tomorrow we want to reuse this upcase with list container then we should be straightforward.

For the upcase example mentioned above, we need a map function similar to Enum.map which map each push to a function before passing to underlying container type.

defmodule Coll do
  @spec map((any -> any), Collect.t()) :: Collect.t()
  def map(func, collect) do
    Collect.build(
      fn -> Collect.init(collect) end,
      fn acc, elem -> Collect.call(collect, func.(elem)) end,
      fn acc, status -> Collect.finish(collect, status) end,
    )
  end
end

Note that map returns Collect.t() as return type so generate_strings happy to accept this new container.

Now we can support multiply without changing generate_numbers

list = generate_strings(
  Coll.map(
    fn str -> String.upcase(str) end,
    list_collect()
  )
)

# => ["FOO ", "BAR"]

Not that we are passing the container as the last argument might look awkward but this is important from a readability point of view which will be obvious when we see more complex example.

Let's implement few more basic operations

def Coll do
  def reduce(func, %Collect{} = collect) do
    Collect.new(
      fn ->
        Collect.init(collect)
      end,
      fn elem, collect ->
        func.(elem, collect)
      end,
      fn collect, status ->
        Collect.finish(collect, status)
      end
    )
  end

  def map(func, %Collect{} = collect) do
    reduce(
      fn elem, collect ->
        Collect.call(collect, func.(elem))
      end,
      collect
    )
  end

  def filter(func, %Collect{} = collect) do
    reduce(
      fn elem, collect ->
        if func.(elem) do
          Collect.call(collect, elem)
        else
          collect
        end
      end,
      collect
    )
  end
end

Equipped with these functions we can compose them however we want

def format_string(label, collect) do
  Coll.map(
    fn str -> "#{DateTime.utc_now()} : #{str}" end,
    collect
  )
end

list =
  generate_strings(
    format_string(list_collect())
  )

# filter strings with length > 3 and write upcase of those strings
string =
  generate_strings(
    Coll.filter(
      fn str -> String.length(str) > 3 end,
      Coll.map(
        fn str -> String.upcase(str) end,
        string_collect()
      )
    )
  )
# => "FOO BAR"

All our functiosn works with Collect type but since elixir already provides Collectable protocol and many types and different libraries already implement it. All we need now is just to bridge the Collect and Collectable protocol for interoperability.

def from_collectable(collectable) do
  Collect.new(
    fn ->
      Collectable.into(collectable)
    end,
    fn {acc, collector_func}, elem ->
      collector_func.(acc, {:cont, elem})
    end,
    fn {acc, collector_func}, status ->
      collector_func.(acc, status)
    end
  )
end

defimpl Collectable, for: Collect do
  def into(%Collect{} = collect) do
    collector_func = fn
      acc, {:cont, elem} ->
        Collect.call(acc, elem)

      acc, :halt ->
        Collect.finish(acc, :halt)

      acc, :done ->
        Collect.finish(acc, :done)
    end

    acc = Collect.init(collect)
    {acc, collector_func}
  end
end

Example

mapset =
  Enum.into(
    [1..100],
    Coll.map(
      fn num -> num * 2 end,
      Coll.from_collectable(MapSet.new())
    )
  )


# => [2, 4, 6, 8 ...]

And for the original example problem, we set out to solve. Which is how to split stream of logs

:ok =
  write_logs(
    stream,
    Coll.filter(
      fn str -> String.start_with?("foo:") end,
      Coll.from_collectable(File.stream!("foo.log"))
    )
  )
